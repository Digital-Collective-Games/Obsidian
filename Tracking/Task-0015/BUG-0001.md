# BUG-0001 — Launched queue agent is killed by the dispatch-activity context

Task: Task-0015. Found by: PASS-0006 / REG-007 live end-to-end (the regression
caught it — exactly its purpose). Severity: blocks the O5 full chain (agent launch).

## Symptom

After a real-UI `Queue=Ready` flip, the always-on consumer correctly NOTICED it
(≤1 min) and DISPATCHED Task-0005 → provisioned an owned worktree
(`...\cdxow\Task-0005-...\w`) → recorded the O6 binding (session id
`309fabd8…`, run/gate state `running`). But the launched claude agent **never ran**:
no transcript at the bound path, no `AGENT-RAN.txt`, for 160s, and zero
launch/error lines in the backend log.

## Diagnosis (proven)

Running the launcher's EXACT command shape manually in that same worktree
(`claude --session-id … -p … --permission-mode bypassPermissions --allowedTools
Read,Write,Edit --output-format json`, with the `claudeChildEnv` overrides) WORKED:
claude ran, `is_error:false`, created `AGENT-RAN.txt`. So claude + the command +
the worktree are fine.

Root cause: `Launcher.Start` builds the process with
`exec.CommandContext(ctx, …)` (`internal/queue/launcher.go:322`), and
`liveLauncher.Launch` passes the **dispatch-activity context**
(`internal/temporalbackend/queuelaunch.go`). The consumer dispatches inside the
Temporal `queue.drain.poll` activity; with `wait=false` the launcher returns
immediately after `cmd.Start()`, the poll activity then completes, its context is
cancelled, and `exec.CommandContext` **kills the child claude process**. The O6
binding is written (BindLaunchedSession) before the kill, so the session id
persists with no transcript — exactly the observed symptom.

## Fix

The launched agent must OUTLIVE the dispatch activity (it is supervised externally
by the watchdog and reclaimed on close, not by the activity context). Launch it
under a detached context (`context.Background()`) instead of the activity ctx, in
`liveLauncher.Launch`. One-line change + comment.

## Remaining / follow-up (not this fix)

- Orphan lifecycle: with a detached context the agent is no longer killed on
  activity completion (correct), but it is also not killed on backend shutdown or
  on worktree reclaim. A bounded task finishes on its own; a long/stuck agent
  should be terminated when its owned lane is reclaimed (close) or on shutdown.
  Track as O5 hardening (kill the launched PID on `cleanupOwnedLane`/shutdown).
- The launcher should LOG the launch (executable, session id, worktree) and any
  `cmd.Start()` error — the silent fire-and-forget made this defect invisible in
  the backend log.

## Verification

Re-run REG-007 end-to-end with the rebuilt binary: real-UI `Queue=Ready` flip →
consumer dispatch → the launched claude agent now persists, writes its transcript
at the bound path, spawns under the worktree, and the bounded task creates
`AGENT-RAN.txt`.
