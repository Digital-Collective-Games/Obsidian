# BUG-0002 — Closed task's worktree never deallocates; consumer poll wedges

Status: **RESOLVED** (PASS-0008, 2026-05-30) — verified live by the full-cycle
deallocate/reuse regression: [PASS-0008/REG-007-FULL-CYCLE-PROOF.md](./Testing/PASS-0008/REG-007-FULL-CYCLE-PROOF.md)
(both worktrees deallocate on auto-close, a 3rd task reuses a freed slot, NO
`is not a working tree` wedge in the backend log). Found by the same regression
(human directive 2026-05-30). Related: [BUG-0001](./BUG-0001.md), O5 orphan-agent
lifecycle ([PASS-0005-AUDIT.md](./Testing/PASS-0005-AUDIT.md)).

## Symptom

Full-cycle regression Phase A (`queue_workers=2`, `OBSIDIAN_AUTO_CLOSE_QUEUED=true`):
- Task-0005's agent announced done (worktree `TASK-STATE.json current_gate="closure"`).
- Consumer auto-closed issue #5 (`auto-close: closed issue #5 ... simulated human closure approval`). ✅
- **But the worktree was NEVER reclaimed**: `/api/v1/worktrees` kept listing Task-0005;
  every subsequent poll errored and the consumer made no further progress (Task-0006's
  lifecycle also stuck):

```
ERROR ... reclaim Task-0005: remove owned worktree: exit status 128:
fatal: 'C:\...\cdxow\Task-0005-...4242049463\w' is not a working tree
WARN  queue-drain poll failed; continuing on next tick ...
```

`git worktree list` showed NO Task-0005 worktree, yet the dir + bootstrap record
persisted → `ListActiveWorktrees` still counted the lane → infinite failing retries.

## Root cause (two compounding defects)

1. **Launched agent not terminated before worktree removal.** The detached agent
   ([BUG-0001](./BUG-0001.md) fix) is still alive (or mid-exit) when `Reclaim` runs;
   on Windows its open handle on the worktree dir makes the first
   `git worktree remove --force` PARTIALLY fail — it unregisters the worktree from git
   and deletes some files (observed: `current_gate` read flipped to `no-file`) but
   leaves a residual directory. The launched PID is never persisted, so reclaim cannot
   terminate the agent.
2. **`cleanupOwnedLane` is non-idempotent** ([service.go:1473](../../backend/orchestration/internal/taskrun/service.go#L1473)).
   It runs only `git worktree remove --force` and returns the error. Once the worktree
   is unregistered, every retry fails with `is not a working tree` forever; the residual
   dir keeps the lane "active", so `DrainOnce` returns an error each poll → the whole
   consumer wedges for the repo.

## Impact

A closed issue's worktree never deallocates (auto-close OR a real human close), the slot
is never freed, and the consumer stops progressing — directly breaking the
allocate → complete → close → deallocate → reuse lifecycle the human requires. Production
risk: a single close that races a live agent permanently wedges the drain loop.

## Fix (PASS-0008)

- **A. Terminate the launched agent before removal.** Persist `LaunchResult.PID`
  (`BindLaunchedSession` → `RepoBinding.LaunchedPID`); in `ReclaimOwnedLane`, before
  cleanup, terminate that PID's process tree best-effort (Windows: verify image is
  `claude.exe` via `tasklist`, then `taskkill /PID <pid> /T /F`; else best-effort kill).
- **B. Make `cleanupOwnedLane` idempotent / self-healing.** On `git worktree remove`
  failure: `git worktree prune` + `os.RemoveAll(OwnedRepoRoot)` with a brief retry
  (Windows lock tolerance); treat a now-absent dir as a successful reclaim. A closed
  issue then ALWAYS deallocates, even after a transient partial failure or an
  already-unregistered ("is not a working tree") residual.

## Verification

The same full-cycle regression must show: both slots allocate → agents announce done →
auto-close → BOTH worktrees deallocate (`/api/v1/worktrees` → 0, slots freed) → a 3rd
queued task REUSES a freed slot. No `is not a working tree` wedge in the backend log.
