# BUG-0003 — Cross-repo Task-ID collision: one repo's closed issue reclaims another repo's live lane (kills its agent + deletes its worktree)

Status: **RESOLVED** (Landing 1, 2026-05-30) — fix verified live + unit-tested:
[Testing/PASS-0010/REG-007-BUG-0003-FIX-PROOF.md](./Testing/PASS-0010/REG-007-BUG-0003-FIX-PROOF.md).
Root cause was proven first by an instrumented repro (below), then fixed by repo-namespacing
the run identity + repo-scoping the lane accounting per
[Design/QUEUE-DRAIN-DURABLE-STATE-REDESIGN.md](./Design/QUEUE-DRAIN-DURABLE-STATE-REDESIGN.md)
(Landing 1). Remaining (Landing 2, named, not part of this bug's closure): move gate/binding +
liveness supervision into the per-run Temporal workflow. Found while running the cap=1
serialization regression
([PASS-0009](./Testing/PASS-0009/REG-007-CAP1-SERIALIZATION-PROOF.md)); this is the
previously-unexplained "first attempt wedge" anomaly, now root-caused. Related:
[BUG-0002](./BUG-0002.md) (the reclaim path itself), O3 registry-driven consumer.

## Symptom

With a multi-repo registry, the always-on queue-drain consumer **tears down a healthy,
running dispatched agent of repo B** (kills its launched process, removes its owned
worktree) when repo A's consumer processes one of repo A's CLOSED issues — because both
issues map to the same Task-ID. The victim repo then re-dispatches and wedges:

```
drain QueueDrainTestbed2: dispatch Task-0001: dispatch blocked:
Dispatch is blocked while this task already owns an active run.
```
…repeated every poll → the consumer makes no further progress for that repo.

## Reproduction (deterministic)

Registry with TWO repos sharing the default runs-root:
- `QueueDrainTestbed` (`queue_workers=2`) — has a **CLOSED** issue **#1**.
- `QueueDrainTestbed2` (`queue_workers=1`) — has an **OPEN/Ready** issue **#1** (dispatched).

Isolated namespace `reg007k`, `LAUNCH_AGENT=true`, shared default runs-root. Instrumented
backend log ([Testing/PASS-0009/evidence/repro-instrumented-log.txt](./Testing/PASS-0009/evidence/repro-instrumented-log.txt)):

```
13:58:39  poll acted dispatched [Task-0001]                       <- QueueDrainTestbed2's OPEN #1
13:59:03  RECLAIM-DEBUG repo=Digital-Collective-Games/QueueDrainTestbed issue #1 closed
          -> RECLAIM taskID=Task-0001 (active lane seen for this repo's service)
13:59:03  RECLAIM-DEBUG ReclaimOwnedLane taskID=Task-0001
          declaredWorktreeRoot=C:\Agent\QueueDrainTestbed
          ownedRepoRoot=...\cdxow\Task-0001-...\w killPID=59336
          (removing this worktree + killing this agent)
13:59:04  ERROR drain QueueDrainTestbed2: dispatch Task-0001: ... already owns an active run
```
PID 59336 (QueueDrainTestbed2's launched agent) confirmed dead afterward. Note the reclaim
runs under `declaredWorktreeRoot=C:\Agent\QueueDrainTestbed` (repo A) yet removes the lane +
kills the agent that belong to `QueueDrainTestbed2` (repo B).

## Root cause (two compounding defects)

1. **Task-ID is not namespaced by repo.** `TaskIDForIssue(n) = fmt.Sprintf("Task-%04d", n)`
   ([consumer.go:109](../../backend/orchestration/internal/queue/consumer.go#L109)) keys
   ONLY on the issue number. In a multi-repo registry, issue numbers are unique only
   *per repo* — so repo A's `#1` and repo B's `#1` both become `Task-0001`. (Every repo has
   an issue #1, so this is the common case, not an edge case.)
2. **Per-repo lane accounting reads a GLOBAL, non-repo-scoped runs-root.**
   `ActiveOwnedLaneTasks → ListActiveWorktrees`
   ([service.go:182](../../backend/orchestration/internal/taskrun/service.go#L182)) scans the
   shared `<runs-root>/taskruns/<taskID>/…/owned-lane-bootstrap.json` for ALL repos and is
   not filtered to the calling service's repo. So repo A's per-repo `Consumer` "sees" repo
   B's `Task-0001` lane as active.

The collision then fires through the consumer's existing terminal-close path
([consumer.go:183](../../backend/orchestration/internal/queue/consumer.go#L183)): repo A's
CLOSED `#1` → `ActionTerminal`; `active[Task-0001]` is true (B's lane, seen via the global
runs-root) → `Reclaim(Task-0001)` → `ReclaimOwnedLane` terminates B's launched PID
([service.go:380](../../backend/orchestration/internal/taskrun/service.go#L380)) and removes
B's worktree. The `RegistryConsumer` iterates repos in order and swallows the per-repo error
([registry_consumer.go:135](../../backend/orchestration/internal/queue/registry_consumer.go#L135)),
so the reclaim is never even surfaced in the aggregate "poll acted" log — only repo B's
follow-on re-dispatch error is.

## Impact

Directly breaks the core O3 multi-repo registry feature (global awareness of N repos): any
repo closing its issue #N will reclaim a *different* repo's live `Task-N` — killing a
running agent and deleting its worktree mid-work. High severity for real multi-repo use.
A related deeper gap (same root cause): if **two** repos both have an OPEN/Ready issue #N,
they collide on the global Temporal run-id `taskrun--Task-000N--active` and on the shared
`taskruns/Task-000N/…` bootstrap path (one clobbers the other) — so even the non-close path
is unsound for colliding issue numbers across repos.

## Why PASS-0009 still passed

The PASS-0009 cap=1 lifecycle was proven on an ISOLATED single-repo registry + dedicated
runs-root, which removes BOTH defects' triggers (no second repo with a colliding closed #1;
no shared runs-root). The dispatch/serialization logic itself is correct; this bug is in
cross-repo identity/accounting, not in the cap logic.

## Proposed fix

- **A (surgical — closes the agent-killer):** repo-scope the consumer's active-lane
  accounting. `ActiveOwnedLaneTasks` should only return lanes whose record
  `DeclaredWorktreeRoot` matches the calling service's `declaredWorktreeRoot` (the data is
  already on every bootstrap record). Keep `ListActiveWorktrees` global for the
  `/api/v1/worktrees` endpoint. This alone stops a repo from reclaiming another repo's lane.
- **B (deeper — full multi-repo correctness):** namespace the GLOBAL task identity by repo —
  the Temporal run-id (`ActiveRunID`), the runs-root artifact path
  (`taskruns/<taskID>/…`), and the lane base — so two repos can each run their own `#N`
  without clobbering run-id/runs-root. Larger blast radius (touches run-id, artifact paths,
  enumeration); warrants its own scoped pass + QA.

Recommended: land **A** now (contained, verifiable by re-running the repro + PASS-0009),
track **B** as a follow-up.
