# BUG-0005 — Eject frees the worktree but leaves the TaskRunWorkflow active/orphaned

- **Task:** Task-0016
- **Opened:** 2026-06-01 (found during the live production-roll Assign→Eject smoke)
- **Severity:** human-surface correctness gap — an operator Eject should stop the run, not
  leave an orphaned active run with no worktree
- **Status:** FIXED (Eject now terminates the run's TaskRunWorkflow; unit-proven for a
  running AND a parked lane). Live re-verify of the strengthened
  [REG-015](../../REGRESSION.md) is the coordinator's redeploy step.

## Symptom (human / operator facing)

After a **successful** in-app/endpoint Eject (`POST /api/v1/worktrees/eject` → HTTP 200; the
worktree returns to idle, the launched agent is killed, and the freed task is dequeued), the
run's Temporal workflow `taskrun--Task-<id>--active` **remains ACTIVE/running** on the
namespace with no bound worktree. Verified live: after ejecting Task-0016's worktree (and
then Destroying the now-idle worktree), `temporal workflow describe -w
taskrun--Task-0016--active` still showed the workflow running. It had to be terminated
manually (`temporal workflow terminate`). So Eject frees the worktree but orphans the run.

## Evidence (live `:4318`, production `default` namespace, smoke)

- Create `CodexDashboard/wt-0002` → Assign `Task-0016` → worktree `allocated`, run
  `taskrun--Task-0016--active` `running`, owned checkout bootstrapped.
- Eject `{run_id: taskrun--Task-0016--active}` → HTTP 200, worktree → `idle`, task dequeued
  (`Queue=Never`, #16 stays open). **But** `temporal workflow describe` showed the workflow
  still `Running` afterward; `GET /api/v1/worktrees` showed no allocation (worktree freed).
- Destroyed the idle worktree (200) → run now fully orphaned (active workflow, no worktree).
- Cleaned up: `temporal workflow terminate -w taskrun--Task-0016--active -n default`.

## Secondary observation

`GET /api/v1/task-runs/taskrun--Task-0016--active` continued to report `status=active
state=running` even AFTER Temporal had terminated the workflow (stale read — the backend
read path does not detect a terminated workflow as gone).

## Expected

Per the Task-0016 Eject contract ("stop the launched agent + clean the checkout +
unbind/**terminate the run** + keep folder + idle + dequeue"), Eject should TERMINATE the
run's Temporal workflow (or move it to a definitive terminal/parked state) when freeing the
worktree, so no active run lingers with no worktree. The run read should reflect a terminated
workflow as not-active.

## Fix (proposed)

- `Service.EjectWorktree` should terminate the run's `TaskRunWorkflow` (the same
  terminate/reclaim the close path uses) when it frees the worktree — not only unbind the
  worktree.
- Make the run read (`GetActiveTaskRun` / `/api/v1/task-runs/<id>`) treat a
  terminated/closed Temporal workflow as not-active (avoid the stale `active` read).
- Then strengthen [REG-015](../../REGRESSION.md) to assert the run is terminated after Eject
  (not just the worktree idle).

## Resolution (FIXED)

Root cause: the per-run `TaskRunWorkflow`
([taskexec.go](../../backend/orchestration/internal/taskexec/taskexec.go)) is a long-running
workflow that loops on signals and only exits on a TERMINAL status
(`completed`/`failed`/`interrupted`). Once the owned-lane execution finishes it sits in
`running` waiting on signals indefinitely. Neither the close path (`ReclaimOwnedLane`) nor
Eject ever terminated it — they only killed the agent PID + reclaimed/freed the worktree. So
the workflow stayed ACTIVE/orphaned (live-observed for Eject).

Fix (mirrors the existing best-effort agent-kill pattern, not a new termination path):

- Added `Runtime.TerminateTaskRun(ctx, runID, reason)` to the runtime interface
  ([types.go](../../backend/orchestration/internal/taskrun/types.go)) and implemented it on
  the Temporal `Backend`
  ([backend.go](../../backend/orchestration/internal/temporalbackend/backend.go)) as a thin
  `client.TerminateWorkflow(ctx, runID, "", reason)` that maps an already-gone run
  (Temporal not-found) to success (idempotent) — the same place the existing
  signal-by-runID methods live.
- `Service.EjectWorktree`
  ([pool.go](../../backend/orchestration/internal/taskrun/pool.go)) now calls
  `s.runtime.TerminateTaskRun(ctx, record.RunID, "ejected by operator")` for the bound run,
  in addition to its existing agent-kill + clean (`reset --hard` + `clean -fdx`) + return-idle
  (folder kept) + dequeue. Best-effort + idempotent, so a re-eject or a parked run still
  ejects cleanly. It never closes the issue and never deletes the folder.
- Run-read staleness: terminating the workflow IS the staleness fix. After Eject the pool
  read (`ListPoolWorktrees` / `GET /api/v1/worktrees`) reclassifies the member to idle
  (`classifyPoolMember` → `GetActiveTaskRun` returns `ErrRunNotFound` for a gone run), and the
  direct read `/api/v1/task-runs/<id>` (`Service.Run` → `GetTaskRun`) can no longer query a
  terminated workflow, so it returns not-found (404) rather than a stale `active/running`. No
  separate read-path change was needed; a focused follow-up note: if Temporal returns a
  terminate-specific query error whose string lacks "not found", the direct read returns 502
  rather than 404 — still NOT a stale `active` claim, so it is left as-is (verifying the exact
  error shape needs a live Temporal harness).

Test: `TestEjectWorktreeTerminatesRunForRunningAndParkedLane`
([pool_test.go](../../backend/orchestration/internal/taskrun/pool_test.go)) — asserts that
after `EjectWorktree` the fake runtime recorded a terminate call for the bound run id AND the
run no longer reads active (the member returns idle with no bound run), for BOTH a running and
a parked lane. The falsifier (an Eject that frees the worktree but never terminates the run —
the live-observed orphan) fails the test.

## Note on scope

Found while making the live human surface actually work (the user's "done = working human
surface" bar). REG-015 (Eject) passed on the validation lane because it only checked the
worktree returned to idle + the dequeue, not the run-workflow lifecycle — the same
clean-conditions gap that let BUG-0003 through. Both are why REG-015/REG-017 are being
hardened to exercise the real post-action state.
