# BUG-0002 — Manual-pool repo-segment mismatch: a Create-seeded worktree is invisible to the queue-drain consumer's pool-draw (REG-007 live blocker)

- **Task:** Task-0016
- **Opened:** 2026-06-01 (LIVE PASS-0009 REG-007 run, isolated `reg007live` lane)
- **Severity:** Blocking for closure of the REG-007 "pool of 1" reinterpretation (seed via Create → consumer draws). The documented Proof-Plan flow cannot complete on the real surface.
- **Status:** OPEN (product defect; reproduced live; not yet fixed)

## Symptom (human / operator facing)

An operator pre-creates an idle pool worktree for a repo via the desktop `WORKTREES`
tab Create control (or `POST /api/v1/worktrees/create {repo}`), then flips that repo's
GitHub issue `Queue=Ready`. The always-on queue-drain consumer **never dispatches into
the Created worktree** — it behaves as if the pool is empty (`dispatched []`), so the
Ready issue waits forever. This contradicts the REG-007 "pool of 1" contract
([REGRESSION.md](../../REGRESSION.md) REG-007 reinterpretation; [TASK.md](./TASK.md)
Proof Plan: "with the pool seeded via Create first").

## Root cause — two services segment the same `ownedLaneRoot` by DIFFERENT keys

The control-plane process runs two kinds of `taskrun.Service`, and they disagree on the
per-repo pool subdirectory ("repo segment") under the shared owned-lane root
(`<TempDir>/cdxow` on Windows):

- **Dashboard control-plane `taskService`** (serves `POST /api/v1/worktrees/create` and
  the `WORKTREES` tab) is built via `taskrun.NewService(cfg.WorktreeRoot, ...)`
  ([cmd/controlplane/main.go:34](../../backend/orchestration/cmd/controlplane/main.go#L34))
  with **no `SetRepoNamespace`**. Its `poolRepoSegment()`
  ([internal/taskrun/pool.go:92](../../backend/orchestration/internal/taskrun/pool.go#L92))
  falls to `repoIdentity()` → `LoadManifest(s.declaredWorktreeRoot)` against
  `cfg.WorktreeRoot`, which is NOT any registry `local_root`, so no manifest entry matches
  and it returns a **hash** of `cfg.WorktreeRoot`: `repo-<hash>`.
  `CreatePoolWorktree(repo)` writes the member under `<ownedLaneRoot>/repo-<hash>/wt-NNNN`
  and stamps the record `Repo` field with the request `repo` arg ("RepoA") — but the
  PHYSICAL segment and the `worktree_id` are `repo-<hash>/wt-NNNN`.

- **Registry-driven consumer per-repo `Service`** is built via
  `taskrun.NewServiceForRepo(repo.LocalRoot, ...)` + `SetRepoNamespace(repo.ID)`
  ([internal/temporalbackend/queuedrain.go:197-201](../../backend/orchestration/internal/temporalbackend/queuedrain.go#L197)).
  Its `poolRepoSegment()` short-circuits to the registry **repo id** ("RepoA"). Its idle
  budget (`countIdlePoolWorktrees` → `ListPoolWorktrees` → `enumeratePoolRecords` of
  `poolRepoRoot()` = `<ownedLaneRoot>/RepoA`) only ever reads the **`RepoA` segment**.

So a Create lands in `<ownedLaneRoot>/repo-<hash>/` while the consumer draws from
`<ownedLaneRoot>/RepoA/`. The two never meet: the consumer's idle count for RepoA is 0,
and an empty pool defers the Ready issue (no auto-create, by design).

This is structurally identical to [BUG-0001](./BUG-0001.md): the **multi-repo dashboard
Service** holds a single `declaredWorktreeRoot` and cannot, on its own, map a per-repo
operation onto the registry repo's identity. BUG-0001 fixed it for the dequeue WRITE
(a registry-resolving provider); the pool segment for Create (and for the `WORKTREES`-tab
Eject/Destroy, which key on the same `worktree_id`/segment) has the same gap.

## Secondary observation (recorded honestly; likely a test-setup confound, NOT a confirmed second bug)

In the FIRST REG-007 attempt, closing `QueueDrainTestbed #1` did not produce a
`reclaimed [Task-0001]` and the deferred `Task-0005` errored
`dispatch Task-0005: ... the repo worktree pool has no idle worktree to draw`
(consumer-backend.log @ 14:14:37). At that point the lane state was **confounded**: a
dashboard-Assigned run (`taskrun--Task-0001--active`, non-namespaced) and a consumer
hand-seeded lane (`taskrun--RepoA--Task-0001--active`) both existed for issue #1, so the
reclaim target was ambiguous. **Later in the same run the consumer's close→reclaim DID
work cleanly** for REG-009: closing `QueueDrainTestbed2 #1` produced
`reclaimed [Task-0001]` and removed only RepoB's checkout (see
[Testing/PASS-0009/reg009-live/REG-009-LIVE-RESULT.txt](./Testing/PASS-0009/reg009-live/REG-009-LIVE-RESULT.txt)).
So the close→reclaim path itself is functional for a cleanly-owned consumer lane; the
first-attempt anomaly is best explained as a confound from the mixed dashboard+consumer
lane setup, not an independent defect. One residual nit observed on reclaim: the freed
pool record's `run_id` is not nulled (the checkout is removed but `worktree-pool.json`
keeps the stale `run_id`); minor, recorded for follow-up, not blocking.

## What this blocks

- **REG-007 (pool-of-1) live, end-to-end** via the documented Create-seed path:
  - PROVEN: real-UI Ready flip commits; consumer dispatches exactly one into a pool
    worktree within ≤1 min of an idle worktree existing; a top-level `claude` agent
    launches and is discoverable; a SECOND Ready issue **waits** (empty pool defers, no
    auto-create — verified: only one `RepoA/wt-0001` member, no `wt-0002`, no Task-0005
    workflow).
  - BLOCKED: seeding that draw via the product **Create** action (the worktree is
    invisible to the consumer), and the **close/eject → free → reuse** leg.

## Required fix (proposed; mirrors the BUG-0001 pattern)

Make the per-repo pool segment registry-resolved on the multi-repo dashboard Service so
a `repo`-keyed Create/Eject/Destroy lands in (and reads) the SAME segment the
registry-driven consumer uses for that repo. Options:

- Resolve the segment from the `repo` argument via the registry inside
  `CreatePoolWorktree(repo)` / the worktree handlers (look up `repo` → registry id →
  segment), instead of from the Service's single `declaredWorktreeRoot`; OR
- Build the dashboard worktree operations through a registry-resolving wrapper (as the
  dequeue provider was), so each per-repo op uses a Service namespaced by the registry
  repo id.

Then re-confirm: a UI Create for RepoA produces a `RepoA/wt-NNNN` worktree that the
consumer draws on the next Ready flip, and close/eject frees it for the deferred issue.
