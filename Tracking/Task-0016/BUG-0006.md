# BUG-0006 — `GET /api/v1/tasks` 502s after a run is TERMINATED → breaks the WORKTREES Assign popup

- **Task:** Task-0016
- **Opened:** 2026-06-01 (found live during the post-[BUG-0005](./BUG-0005.md) Eject re-verify)
- **Severity:** closure-blocking human-surface break — the WORKTREES tab's Assign popup is
  backed by `GET /api/v1/tasks`; once that 502s the popup blanks and an operator cannot assign
- **Status:** FIXED (readTask now tolerates a terminated/gone run at the refreshRun site too; a
  terminated workflow maps to `ErrRunNotFound` so the direct read returns 404, not 502/stale).
  Unit-proven, including the live failing shape via a falsifier. Live redeploy + re-verify of
  the new [REG-018](../../REGRESSION.md) is the coordinator's step.

## Symptom (human / operator facing)

On the live backend, `GET /api/v1/tasks` returns **HTTP 502** with body
`{"error":"task run not found"}` once a task's run workflow has been TERMINATED. This breaks
the WORKTREES tab's Assign popup (its source is `/api/v1/tasks`). It is NOT the
[BUG-0003](./BUG-0003.md) missing-`TASK-STATE.json` case (that is fixed) — it is the run-read
path for a *terminated* run on a still-non-terminal task.

**This recurs after EVERY Eject.** The [BUG-0005](./BUG-0005.md) fix made Eject terminate the
run's `TaskRunWorkflow` while the task's `TASK-STATE.json` stays non-terminal, so the next
`ListTasks` → `readTask` queries the terminated run and 502s. BUG-0005's own resolution note
predicted exactly this read-path gap (see its "if Temporal returns a terminate-specific query
error whose string lacks 'not found', the direct read returns 502 rather than 404").

## Confirmed root cause (exact failing path)

`readTask` ([service.go](../../backend/orchestration/internal/taskrun/service.go), L1148) for a
non-terminal task with a runtime had TWO run-read sites with INCONSISTENT not-found handling:

1. `GetActiveTaskRun` (L1185) — **already tolerated** `ErrRunNotFound` (L1187 returns the view).
2. `refreshRun` (L1199) — **did NOT tolerate** it: `return TaskView{}, err` on any error (old
   L1200-1201).

For a TERMINATED workflow, `Backend.GetTaskRun`
([backend.go](../../backend/orchestration/internal/temporalbackend/backend.go) L172) `QueryWorkflow`
can SUCCEED (querying a closed workflow's last state), so `GetActiveTaskRun` returns a run with
no error and the L1185 tolerance never fires. Then `refreshRun` →
`UpdateTaskRun` ([backend.go](../../backend/orchestration/internal/temporalbackend/backend.go) L247)
sees the run is gone and returns `ErrRunNotFound` (either its `GetTaskRun` query now returns
not-found, or `readUpdatedTaskRun` falls through to `getClosedTaskRunResult` whose
`workflowRun.Get` returns a `*temporal.TerminatedError`). That un-tolerated `ErrRunNotFound`
propagated up `readTask` → `ListTasks`, and the handler at
[mux.go](../../backend/orchestration/internal/httpapi/mux.go) L123-125 maps any `ListTasks`
error to **502** (unlike the detail handlers, `ListTasks` does no `"not found"`→404 translation).
The 502 body being EXACTLY `task run not found` (= the `ErrRunNotFound` message,
[types.go](../../backend/orchestration/internal/taskrun/types.go) L11) is the smoking gun that
the sentinel itself surfaced un-tolerated from the refreshRun site.

A second consistency gap: `getClosedTaskRunResult`'s `workflowRun.Get` on a *terminated* (not
completed) workflow returns a `*temporal.TerminatedError` whose message lacks "not found", so
`isTemporalNotFound` did NOT classify it as gone → the direct read `/api/v1/task-runs/<id>`
([Service.Run](../../backend/orchestration/internal/taskrun/service.go) L800) could return 502
(or a stale active) instead of 404 for a terminated run.

## Fix

Surgical, additive, consistent with the existing `isTemporalNotFound`→`ErrRunNotFound` pattern:

1. **readTask refreshRun tolerance**
   ([service.go](../../backend/orchestration/internal/taskrun/service.go) ~L1199): tolerate
   `ErrRunNotFound` from `refreshRun` the SAME way the `GetActiveTaskRun` site already does —
   `return view, nil` (the task view with `CurrentStory = no_active_run`), instead of
   `return TaskView{}, err`. A terminated/gone run now reads as "no active run" at BOTH run-read
   sites in this function, never a 502.

2. **Terminated workflow → `ErrRunNotFound` mapping**
   ([backend.go](../../backend/orchestration/internal/temporalbackend/backend.go), `isTemporalNotFound`):
   also treat `temporal.IsTerminatedError(err)` as "gone". This is the single place every
   signal/query/closed-result site already funnels through to produce `ErrRunNotFound`, so a
   terminated-workflow query/`getClosedTaskRunResult` now consistently surfaces the sentinel →
   the direct read `/api/v1/task-runs/<id>` returns 404 (not 502/stale active), AND `readTask`
   tolerates it. Added the `go.temporal.io/sdk/temporal` import.

No task was marked complete to dodge this — the task state is legitimately non-terminal while
its run is gone; the fix makes the *run read* report "no active run", not the *task* complete.

## Tests (unit)

All in [service_test.go](../../backend/orchestration/internal/taskrun/service_test.go) and
[backend_test.go](../../backend/orchestration/internal/temporalbackend/backend_test.go):

- `TestListTasksToleratesTerminatedRun` — table test, BOTH failing shapes via a
  `terminatedRunRuntime` (embeds the standard fakeRuntime):
  - `not_found_at_get_active_run`: `GetActiveTaskRun` returns `ErrRunNotFound`.
  - `not_found_at_refresh_run`: `GetActiveTaskRun` returns a TERMINAL run, then
    `refreshRun`'s `UpdateTaskRun` returns `ErrRunNotFound`.
  Both must yield NO error and the task present with `no_active_run`.
- `TestRunForTerminatedRunReturnsErrRunNotFound` — a direct `Service.Run` for a gone/terminated
  run returns `ErrRunNotFound` (so the API returns 404, not 502).
- `TestIsTemporalNotFoundTreatsTerminatedWorkflowAsGone` — a `*temporal.TerminatedError` maps to
  not-found; an unrelated error does not.

**Falsifier verified:** with the refreshRun tolerance removed, `not_found_at_refresh_run` FAILS
with the exact live error `task run not found`, while `not_found_at_get_active_run` still passes
(it was already tolerated). This reproduces and pins the live 502.
