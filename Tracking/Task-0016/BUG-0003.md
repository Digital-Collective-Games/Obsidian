# BUG-0003 — `GET /api/v1/tasks` 502s (Assign popup breaks) when any task dir lacks `TASK-STATE.json`

- **Task:** Task-0016
- **Opened:** 2026-06-01 (found during the live production roll, testing the Assign Task button)
- **Severity:** Blocking the [FE] Assign popup on the live human surface (the popup's data source 502s)
- **Status:** FIXED (commit pending) — robustness fix + regression test; needs the service-lane republish to take effect live

## Symptom (human-facing)

On the live backend the desktop `WORKTREES` tab's **Assign Task** popup cannot load its
open-task list: `GET /api/v1/tasks` returns **HTTP 502** with
`read C:\Agent\CodexDashboard\Tracking\Task-0014\TASK-STATE.json: ... The system cannot
find the file specified.` Create/Destroy/list/repos all work; only the Assign popup
(which reads `GET /api/v1/tasks`) is broken.

## Root cause

`ListTasks` ([service.go:654](../../backend/orchestration/internal/taskrun/service.go#L654))
iterates every `Tracking/Task-*` dir and calls `readTask` → `loadTask`, which **hard-failed
the entire list** if any single task's `TASK-STATE.json` was missing
([loadTask, service.go:1260](../../backend/orchestration/internal/taskrun/service.go#L1260)).
**Task-0014** (1 of 16 task dirs) has no `TASK-STATE.json`, so the whole endpoint 502s — and
the Assign popup (source = `GET /api/v1/tasks`, pinned by the task) goes with it. This is a
pre-existing fragility, exposed by the Task-0016 Assign popup being the live consumer of the
endpoint against the real registry/Tracking state. (The validation-lane REG-014 used a
throwaway registry where every task had a state file, so it didn't surface there.)

## Fix

Make `loadTask` tolerate a missing `TASK-STATE.json`: a `os.IsNotExist` on the state file
surfaces the task with a **default/unknown state** (TaskID derived from the dir name) instead
of erroring; other read/decode errors still fail. The TASK-STATE.json evidence ref is only
added when the file exists. One stateless task no longer breaks the whole list / Assign popup
(it appears with an unknown state, truthfully — its state file genuinely doesn't exist; no
state is fabricated).

- `backend/orchestration/internal/taskrun/service.go` `loadTask`.
- Test: `TestListTasksToleratesMissingTaskState` (service_test.go) — a Tracking root with one
  task missing `TASK-STATE.json`; `ListTasks` returns no error and the stateless task is still
  listed with its id.

## Verification

- `go build ./...`, `gofmt` clean; `go test ./internal/taskrun/` (package) green incl. the new
  test.
- LIVE: after rebuilding + republishing the service-lane release and restarting `:4318`,
  `GET /api/v1/tasks` returns 200 and the Assign popup populates.
