## Symptom (human-facing)

An operator clicking **Eject** (or the standalone **Dequeue**) on the desktop `WORKTREES`
tab returns the worktree to idle, but the freed task's issue is **not** dequeued
(`Queue` stays `Ready`). If that repo's queue-drain consumer is running, the task is
**re-dispatched** on the next poll — the exact "bounce-back" UPDATE 2 was designed to
prevent. The dequeue is a silent no-op on the dashboard control plane.

## Evidence

- The production control plane builds one `taskService` and hands it to the mux but never
  injects a dequeue provider: [`cmd/controlplane/main.go:51`](../../backend/orchestration/cmd/controlplane/main.go#L51)
  (`httpapi.NewMux(cfg, service, taskService, backend)`); no `SetDequeueProvider` call exists
  in `main.go`.
- `SetDequeueProvider` is wired ONLY on the per-repo queue-drain `Service`
  ([`temporalbackend/queuedrain.go:205`](../../backend/orchestration/internal/queue/queuedrain.go#L205)),
  a **different** `Service` instance from the mux's `taskService`.
- With no provider, the write is a no-op: [`taskrun/pool.go:507`](../../backend/orchestration/internal/taskrun/pool.go#L507)
  (`if s.dequeueProvider == nil { return nil }`). Both the standalone dequeue endpoint
  ([`httpapi/mux.go:298`](../../backend/orchestration/internal/httpapi/mux.go#L298) →
  `taskService.DequeueTask`) and Eject's dequeue ([`pool.go:676`](../../backend/orchestration/internal/taskrun/pool.go#L676))
  therefore no-op in production.
- Honestly flagged by the PASS-0008 leader (PASS-0008 NOTES, caveat #1).

## Root cause

The dequeue **mechanism** (provider write `Queue→Never`, the Service seam, Eject coupling)
is correct and unit-proven (PASS-0004/0005 against a fake provider; the mux test injects a
fake at `worktrees_pool_test.go:218`). The gap is **production wiring**: the dashboard
control-plane `Service` — which spans all registry repos and serves the `WORKTREES` tab —
is never given a real `DequeueProvider`. (The per-repo queue-drain services have one, but
the dashboard does not use those instances.)

## Required fix

Wire a real `DequeueProvider` into the control-plane `taskService` in
[`cmd/controlplane/main.go`](../../backend/orchestration/cmd/controlplane/main.go) so that
in-app Eject/Dequeue actually writes `Queue=Never` to the **correct repo's** task provider:

- The dashboard service is multi-repo, so the injected provider must resolve the worktree's
  pool repo → that repo's `task_provider` GitHub slug (from the registry / `REPO-MANIFEST.json`)
  and write to the right GitHub repo via the existing `ghQueueProvider.DequeueIssue(repo, number)`
  path (the `DequeueIssue(repo, number)` signature already takes the repo). Build it the same
  way the queue-drain provider is built ([`NewGitHubQueueProvider`](../../backend/orchestration/internal/queue/queuedrain.go#L189)).
- Idempotent (already `Never` ⇒ no-op); never closes the issue (human-only closure preserved);
  safe no-op when a worktree has no provider-backed task.
- Keep the test lanes safe: the fix must target the issue the OPERATOR is ejecting (the real
  task), which is correct behavior; regression/test runs continue to use throwaway testbed
  repos on the isolated lane, never a production-owned queue.

## Resolution (fix-pending-live-verification)

**Root cause confirmed.** The dashboard control-plane `taskService`
([`cmd/controlplane/main.go`](../../backend/orchestration/cmd/controlplane/main.go)) was
built via `taskrun.NewService` and never given a `DequeueProvider`, so
[`pool.go` `DequeueTask`](../../backend/orchestration/internal/taskrun/pool.go) returned
early (`s.dequeueProvider == nil`) and the in-app Eject/Dequeue `Queue→Never` write was a
silent no-op. The dequeue mechanism + provider write were already correct and unit-proven
(PASS-0004/0005); only the production wiring on the multi-repo dashboard service was
missing. Confirmed by inspection (no `SetDequeueProvider` in `main.go`; the only call was
on the per-repo queue-drain `Service` in queuedrain wiring).

**The fix (multi-repo resolution path).** The dashboard `Service` spans every registered
repo, and `ghQueueProvider.DequeueIssue(repo, number)` routes the `gh api` write by its
`repo` argument but resolves the org `Queue` field id against its construction owner — so a
single fixed provider is wrong for repos in different orgs, and Eject must target the
*ejected* worktree's repo, not one hardcoded repo. The fix is a registry-backed resolving
`DequeueProvider`:

- New `registryDequeueProvider` + `NewControlPlaneDequeueProvider(registryPath)` in
  [`temporalbackend/queuedrain.go`](../../backend/orchestration/internal/temporalbackend/queuedrain.go):
  on each `DequeueIssue(repo, number)` it resolves `repo` (a registry **id** like
  `obsidian`, or an already-resolved `owner/name` task_provider slug) against the registry
  to that entry's `task_provider.repo` slug, lazily builds + caches a real
  `ghQueueProvider` for that slug (built exactly like the queue-drain read provider via
  `queue.NewGitHubQueueProvider`, so each repo's org field-id resolution is correct), and
  delegates the write. A repo with no GitHub provider configured (unknown repo,
  non-`github_issues` entry, slug not in the registry, or empty registry path) is a **safe
  no-op** — it never crashes. It only delegates `DequeueIssue`, so it **never** closes the
  issue.
- [`cmd/controlplane/main.go`](../../backend/orchestration/cmd/controlplane/main.go):
  injects it once —
  `taskService.SetDequeueProvider(temporalbackend.NewControlPlaneDequeueProvider(cfg.RegistryPath))`.
- [`pool.go` `EjectWorktree`](../../backend/orchestration/internal/taskrun/pool.go): now
  passes the ejected worktree's own `record.Repo` to `DequeueTask` (instead of the removed
  `dequeueRepo()` which returned `""`), so the multi-repo dashboard writes `Queue=Never` to
  the **correct** repo. The standalone dequeue endpoint already passes the request `repo`.
  Idempotent and folder-keeping behavior unchanged; the issue is never closed.

**Verification (unit/integration; no real GitHub).**

- `TestControlPlaneDequeueResolvesRepoToCorrectProvider` /
  `TestControlPlaneDequeueSafeNoOpWhenNoProvider` /
  `TestControlPlaneWiringRoutesServiceDequeueToResolvedRepo`
  ([`temporalbackend/queuedrain_dequeue_test.go`](../../backend/orchestration/internal/temporalbackend/queuedrain_dequeue_test.go)):
  the resolving provider maps `obsidian`→`gregsemple2003/obsidian` and a slug straight
  through to the other repo, routes the `Queue→Never` write to the correct repo + issue
  number through a recording provider, fatals on any `CloseIssue`, and is a safe no-op for
  an unknown/no-provider repo and an empty registry. The Service+provider composition
  (`taskrun.NewService` + `SetDequeueProvider(NewControlPlaneDequeueProvider)`) — the exact

  ([`httpapi/worktrees_pool_test.go`](../../backend/orchestration/internal/httpapi/worktrees_pool_test.go)):
