# BUG-0004 — Flaky test: `TestDashboardEjectAndDestroyResolveRegistryIDSegment` (git "Could not parse object")

- **Task:** Task-0016
- **Opened:** 2026-06-01 (observed running the full Go suite during the live roll)
- **Severity:** test-quality (flaky) AND a latent multi-repo PRODUCT correctness bug in Eject's
  baseline resolution (see Resolution); fixing the product path fixes the flakiness.
- **Status:** FIXED. `go test -count=5 ./internal/taskrun/` green (210s, stable); full
  `go test ./...` green.

## Symptom

`go test ./...` (and `go test -count=3 ./internal/taskrun/ -run TestDashboardEject...`)
intermittently fails:
`eject clean worktree RepoA/wt-0001: reset owned lane to <hash>: exit status 128: fatal:
Could not parse object '<hash>'.` The same test **passes** under package-only
`go test -count=1 ./internal/taskrun/`. So the failure is order/isolation-dependent, not a
product defect (the live eject + the live REG-015 dequeue both work).

## Root cause (hypothesis)

Test-fixture git isolation in
[`pool_segment_test.go`](../../backend/orchestration/internal/taskrun/pool_segment_test.go):
the baseline commit the eject path resets the worktree to (`git reset --hard <baseline>`) is
not reliably present in the test worktree's git object DB under certain run orderings /
parallelism / `-count>1`. The fixture likely captures a baseline commit that another
subtest's git operations (or a reused temp dir) leaves unresolvable.

## Impact

Intermittent red suite — exactly the kind of flakiness that wastes a reviewer's time. Not a
product bug.

## Fix (proposed)

Make the eject fixture deterministic: ensure the worktree's baseline commit is materialized
in (or fetched into) the worktree's object DB, and isolate the fixture git repo per subtest
(no shared/reused git dir across `-count`/parallel runs). Re-run `go test -count=5 ./...` to
confirm stability.

## Root cause (CONFIRMED) — a real multi-repo product bug, surfaced as flakiness

This was NOT only a fixture-isolation problem; it was a latent product bug in
`Service.EjectWorktree`
([pool.go](../../backend/orchestration/internal/taskrun/pool.go)). On the MULTI-REPO dashboard
control plane, a pool worktree for `RepoA` is a `git worktree` of RepoA's own `local_root`,
added from RepoA's baseline commit (`CreatePoolWorktree` correctly resolves the repo's own
local_root via the registry). But `EjectWorktree` resolved its reset baseline from
`gitRevision(s.declaredWorktreeRoot)` — the DASHBOARD Service's UNRELATED declared root, NOT
RepoA. It then ran `git reset --hard <dashRoot's HEAD>` inside RepoA's worktree.

The fixture (`TestDashboardEjectAndDestroyResolveRegistryIDSegment`) builds `dashRoot` and
`repoLocalRoot` from identical content via `writeGitTaskTrackingRoot` (same author/email,
same files). A git commit hash is content-addressed over tree + author/committer timestamp
(whole seconds). When both repos are committed within the SAME second they get the SAME commit
hash, so `git reset --hard dashHEAD` accidentally succeeds (the object exists by collision).
When the two commits straddle a 1-second boundary (timing/ordering/`-count>1`/parallelism
dependent) the hashes differ, `dashHEAD` is not in RepoA's object DB, and the reset fails:
`reset owned lane to <hash>: fatal: Could not parse object '<hash>'`. That is the observed
order-dependent flakiness.

## Fix (DONE)

`EjectWorktree` now resolves the baseline against the ejected MEMBER's OWN repo, mirroring
`CreatePoolWorktree`: prefer `registryRepoLocalRoot(record.Repo)` (the registry-resolved
local_root the worktree was added from), falling back to `declaredWorktreeRoot` for the
single-repo control plane / per-repo consumer (where they are the same repo). The reset now
always targets the correct repo's HEAD, which is deterministically present in the worktree's
object DB — fixing both the flaky test and the latent multi-repo product correctness gap. No
fixture change was needed.

Verification: `go test -count=5 ./internal/taskrun/` green (stable, 210s); full
`go test ./...` green; the same fix path is also exercised by
`TestEjectWorktreeTerminatesRunForRunningAndParkedLane` (uses a real git pool repo).
