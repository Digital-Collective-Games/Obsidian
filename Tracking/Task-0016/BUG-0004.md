# BUG-0004 — Flaky test: `TestDashboardEjectAndDestroyResolveRegistryIDSegment` (git "Could not parse object")

- **Task:** Task-0016
- **Opened:** 2026-06-01 (observed running the full Go suite during the live roll)
- **Severity:** test-quality (flaky); the PRODUCT eject works (live REG-015) — TEST-only
- **Status:** OPEN (tracked; fix pending)

## Symptom

`go test ./...` (and `go test -count=3 ./internal/taskrun/ -run TestDashboardEject...`)
intermittently fails:
`eject clean worktree RepoA/wt-0001: reset owned lane to <hash>: exit status 128: fatal:
Could not parse object '<hash>'.` The same test **passes** under package-only
`go test -count=1 ./internal/taskrun/`. So the failure is order/isolation-dependent, not a
product defect (the live eject + the live REG-015 dequeue both work).

## Root cause (hypothesis)

Test-fixture git isolation in
[`pool_segment_test.go`](../../backend/orchestration/internal/taskrun/pool_segment_test.go):
the baseline commit the eject path resets the worktree to (`git reset --hard <baseline>`) is
not reliably present in the test worktree's git object DB under certain run orderings /
parallelism / `-count>1`. The fixture likely captures a baseline commit that another
subtest's git operations (or a reused temp dir) leaves unresolvable.

## Impact

Intermittent red suite — exactly the kind of flakiness that wastes a reviewer's time. Not a
product bug.

## Fix (proposed)

Make the eject fixture deterministic: ensure the worktree's baseline commit is materialized
in (or fetched into) the worktree's object DB, and isolate the fixture git repo per subtest
(no shared/reused git dir across `-count`/parallel runs). Re-run `go test -count=5 ./...` to
confirm stability.
