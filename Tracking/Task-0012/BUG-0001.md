# BUG-0001: Reconcile/sync report false text_conflict from gh UTF-8 mis-decode under Windows PowerShell 5.1

## Status

Fixed. Verified by a clean reconcile run after the fix.

## Summary

A closeout reconcile re-run on 2026-05-28 reported a `text_conflict` for
`Task-0003` even though the committed handoff
([HANDOFF-GH.md](./HANDOFF-GH.md)) and the prior reconcile artifacts claimed a
clean reconcile (`difference_count: 0`, `conflict_count: 0`).

Root cause: the obsidian-operator scripts capture native `gh` stdout with
`& gh ... 2>&1 | Out-String` (and direct `gh ... --json` assignments) without
forcing UTF-8 console decoding. Under Windows PowerShell 5.1, which is the
interpreter the scripts run under via `powershell -NoProfile`,
`[Console]::OutputEncoding` defaults to a non-UTF-8 code page (CP850/CP1252 on
this machine). `gh` emits UTF-8 JSON, so the smart-quote bytes for
`“total tokens”` (`E2 80 9C` / `E2 80 9D`) were mis-decoded into
mojibake (`ÔÇ...`) before the body comparison ran.

This produced a false `text_conflict`: the locally rendered body had the correct
smart quotes, but the script's `gh issue view` readback decoded the same
GitHub bytes as mojibake.

## First Concrete Disagreement

- Expected: closeout reconcile reports `difference_count: 0`, `conflict_count: 0`.
- Observed: `difference_count: 1`, `conflict_count: 1`, a `Task-0003`
  `text_conflict` with `title_matches: true`, `body_matches: false`, and
  `remote_hash_matches_local: true`.

The embedded-marker SHA matching while the visible body diverged is the tell:
the marker SHA was computed at push time from correct bytes, but the readback
text was corrupted on decode.

## Narrowing Evidence

- `gh issue view --repo Digital-Collective-Games/Obsidian 3 --json body -q .body`
  under pwsh 7 (`[Console]::OutputEncoding` = UTF-8 / CP65001) returns the
  correct smart quote `“total tokens”`. No mojibake.
- The same command under Windows PowerShell 5.1 (`[Console]::OutputEncoding` =
  CP850) returns mojibake: `has smartquote U+201C: False`, `has mojibake: True`.
- The reconcile-generated `Task-0003-REMOTE-BODY.md` contains the mojibake bytes,
  confirming corruption happened on the `gh` read path, not on file write
  (`Write-Utf8NoBom` writes the already-corrupted string faithfully).
- GitHub's stored issue body is correct because
  `Sync-TaskToGitHubIssue.ps1` writes the body to GitHub from a UTF-8 file via
  `gh issue edit --body-file`. Only the readback/compare path was lossy.

## Affected Code

- `skills/obsidian-operator/scripts/Reconcile-TaskGitHubState.ps1`
  (`Invoke-GhJson`, remote `gh issue view` readback)
- `skills/obsidian-operator/scripts/Sync-TaskToGitHubIssue.ps1`
  (`gh issue view` readback used for live conflict checks and final readback)
- `skills/obsidian-operator/scripts/Bootstrap-TaskGitHubIssues.ps1`
  (`gh api` readback used for existing-issue detection)

The earlier handoff "UTF-8 fix" only added `-Encoding UTF8` to local file reads
(`Get-Content`). It did not address native `gh` stdout decoding, which is a
separate seam that only bites under a non-UTF-8 console code page.

## Fix

Force UTF-8 console input/output encoding at the top of each script, after
`$ErrorActionPreference`, so native `gh` output decodes as UTF-8 regardless of
the host console code page:

```powershell
try {
    [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
    [Console]::InputEncoding = [System.Text.UTF8Encoding]::new($false)
} catch {
}
$OutputEncoding = [System.Text.UTF8Encoding]::new($false)
```

This keeps the existing local-file `-Encoding UTF8` reads as durable script
behavior and adds the missing gh-stdout decode guard.

## Verification

- Re-ran `Reconcile-TaskGitHubState.ps1 -DryRun` under `powershell -NoProfile`
  (Windows PowerShell 5.1) after the fix.
- Result: `difference_count: 0`, `conflict_count: 0`.
- Proof: [Testing/TaskGitHubReconcile/RECONCILE-DIFFERENCES.md](./Testing/TaskGitHubReconcile/RECONCILE-DIFFERENCES.md)
  and [Testing/TaskGitHubReconcile/RECONCILE-RESULT.json](./Testing/TaskGitHubReconcile/RECONCILE-RESULT.json).

## Notes

- No GitHub write was needed to resolve the conflict because the GitHub body was
  already correct. The conflict was entirely a readback/compare decoding defect.
- This is the canonical regression guard for PowerShell `gh` UTF-8 handling in
  this repo's task-provider scripts.
